### Looks bad but fine

- `combat.service.ts` `initiate*` méthodes (5 helpers `loadOwnedVillage`/`verifyAndDeductUnits`/`calculateExpeditionTiming` déjà factorisés) — pas de duplication réelle, seulement la cohabitation dans le service ; un split par fichier réduirait peu la dette globale et compliquerait DI.
- `applyStrategyStorageBonus` (resources.service.ts:195) — appelé 2× depuis `updateProduction` et `updateStorageLimit`, mais c'est déjà une méthode privée centralisée ; pas de duplication réelle, juste deux callers.
- `crowns.service.ts updateProduction` — patterns similaires à `resources.service` mais nature différente (compteur globale, pas per-village), pas un candidat partage.
- `recursive getResources` (resources.service.ts:55) — apparent code smell mais conscious : un seul retry après catch-up volontaire, coût négligeable.

### Thème choisi

**D3 — Collapse triple WOOD/STONE/IRON projection into typed resource-key iteration**.

Pourquoi :
- Triple `WOOD/STONE/IRON` cohabite à 3 endroits :
  1. `resources.service.ts fetchBuildingRates` (lignes 235–271, 37 L).
  2. `resources.service.ts updateProduction` + `calculateCurrentResources` :
     déclaration `woodRate/stoneRate/ironRate`, calcul `woodGain/stoneGain/ironGain`,
     puis `Math.min(stock + floor(gain), cap)` × 3 — 2 occurrences quasi-identiques.
  3. `strategy/village-strategy.service.ts getProductionRatesForStrategy` (lignes
     370–410, 40 L) — même triple, sauf qu'il utilise `getProductionRate` async
     (1 `getConfig` par type, donc 3 fetches) au lieu du `computeProductionRate`
     synchrone : N+1 silencieux fixé au passage.
- `RESOURCE_BUILDING_TYPES = ['WOOD','STONE','IRON'] as const` déjà exporté
  par `packages/shared/src/resources/production.ts`, ainsi que `findBuildingByType`
  — la projection devient trivialement itérable.
- Risque très faible : helpers purs, comportement strictement préservé
  (test existant `resources.service.spec.ts` couvre les invariants — pas d'appel
  `computeProductionRate` pour les bâtiments absents, `getConfig` une fois,
  forward de `strategy`).
- Bénéfice :
  - resources.service.ts : -64 L brutes (-99 vs +35 nettes).
  - village-strategy.service.ts : -25 L brutes (-37 vs +12).
  - N+1 latent côté `getProductionRatesForStrategy` éliminé.
  - Ajout d'un futur resource type devient mécanique (1 entrée dans la table).

## Refactor (Phase 2)

### Plan

1. Nouveau `battleforthecrown-backend/src/modules/resources/resource-rate-projection.ts` :
   - `interface ResourceTriple { wood; stone; iron }` (numérique).
   - `projectResourceRates(buildings, getRate)` — itère sur `RESOURCE_BUILDING_TYPES`,
     applique `findBuildingByType` (préserve la sémantique « 0 si absent »).
   - `applyResourceCatchup(stock, ratesPerMin, elapsedMinutes, cap)` — formule
     `Math.min(stock.x + Math.floor(rate.x * minutes), cap)` × 3.
2. Refactor `resources.service.ts` :
   - `fetchBuildingRates` (37 L) → 5 L (1 `getConfig`, 1 `projectResourceRates`).
   - `updateProduction` : remplace les 6 var locales + spread dans `tx.update`.
   - `calculateCurrentResources` : remplace les 6 var locales + retour ×3.
   - Drop import `findBuildingByType` (plus utilisé dans le service).
3. Refactor `strategy/village-strategy.service.ts getProductionRatesForStrategy` :
   - `getBuildingLevel` × 3 + `getProductionRate` × 3 → 1 `getConfig` +
     `projectResourceRates(_, computeProductionRate)` + scale ×60.
   - Imports : `projectResourceRates` ajouté ; `BUILDING_TYPES.WOOD/STONE/IRON`
     plus utilisés ici (toujours utilisés ailleurs dans le fichier).
4. Spec dédié `resource-rate-projection.spec.ts` : 6 cas couvrant le contrat
   pur (callback non-invoqué si bâtiment absent, type/level forwardés,
   cap appliqué, elapsed=0 inerte).
