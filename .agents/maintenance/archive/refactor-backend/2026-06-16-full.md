## Thème : U2 — Extraction des helpers de report builder dans les workers combat

### Contexte

`combat.worker.ts` (1780 L) et `return.worker.ts` (370 L) dupliquaient deux patterns à l'identique dans la couche d'émission des reports (reinforcement / caravan arrival / caravan recall return) :

1. **Snapshot village pour embedding dans un report** — `tx.village.findUnique({ where: { id }, select: { id, name, x, y, userId } })`. 4 sites : combat.worker `handleReinforcementArrival` (×2 origin+host), `handleCaravanArrival` (×1 origin) ; return.worker caravan return recall (×2 origin+target — target sans `userId`).
2. **Dédup des destinataires inbox** — `[...new Set([userA, userB].filter((id): id is string => Boolean(id)))]`. 2 sites combat.worker (reinforcement + caravan arrival).

Aucun helper partagé. Le pattern est suffisamment fixe pour qu'une dérive (oubli d'un champ, error message incohérent) soit invisible à la review.

### Looks bad but fine

- **`combat.worker.ts` 1780 L** : couvre 5 kinds × 2 phases (attack/scout/reinforce/caravan/recall × arrivée/retour). Découpé en handlers privés par kind. Splitter par module disperserait le contexte d'orchestration partagé (outbox + power + rankings + outboxResource…).
- **`event-outbox.service.ts` 853 L** : path critique realtime, durci par PR bf7bf72 (prefetch cache concurrent). Complexité intentionnelle ; touch sans cas d'usage perdu est risqué.
- **`combat.service.ts` 1287 L** : 6 méthodes `initiate*` + orchestration. Surface publique cohérente. Split découperait la API contract.
- **`conquest.service.ts` 644 L** : single bounded context (capture lifecycle).
- **`barbarian-seeding.service.ts` 358 L** : runtime placement, déjà découpé en stratégies de chunk.
