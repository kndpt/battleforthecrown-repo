## Refactor (Phase 2)

**File touched**: `battleforthecrown-backend/src/modules/combat/combat.service.ts` only.

1. `initiateAttack` now calls `this.resolveTargetVillage(tx, worldId, dto)` instead of branching inline on `dto.targetKind`. `defenderUserId` becomes `target.userId` (null for barbarians per Village schema). The rest of the body reads `target.id`, `target.x`, `target.y`.
2. `snapshotDefenderKingdomPowers` return type gains `primaryValue: number | null`, computed inside the helper. `initiateAttack` reads `defenderPowerSnapshot.primaryValue` instead of re-deriving from `values[primaryUserId]`.

Behavior deltas (validation tightening, not loosening):
- Attack now rejects PLAYER_VILLAGE targets where the live village row has `isBarbarian = true` (previously allowed). Frontends always send the matching kind, so no live impact expected.
- Attack now rejects mismatched `targetX`/`targetY` against the live row (previously didn't check). Live coordinates are immutable so the only way this trips is a stale client cache, which should be a no-op for honest clients.
- The combat outbox/expedition writes are byte-identical for the happy path: `target.id` = `targetRefId`, `target.x` = `targetX`, `target.y` = `targetY`, `target.userId` = `defenderUserId`.

Net diff: −46 / +22 lines, single file.
