### Looks bad but fine

- **`combat.controller.ts` 349L / 4 report kinds** : CRUD-shaped (Get list / Get :id / Patch read / Delete), factoriser via mixin/decorator serait moins lisible que la duplication actuelle.
- **`caravan-report.service.ts` 98L** : le pattern `findCaravanEntry` est déjà extrait — sert de référence pour ce run, ne pas y toucher.
- **`map-marker.service.ts` 191L** : advisory lock via `pg_advisory_xact_lock` sur `(userId, worldId)` + upsert avec catch P2002 → CAP-safe sous concurrence. Volontaire.
- **`OutboxPublisher` 121L / 5 méthodes** : E1 documenté comme sweep progressif — un run dédié à l'unification serait ~40 callsites, trop divergent pour un thème unique.
- **`retention.service.ts:456-464` `$transaction` wrap 2 updateMany** : cohérent atomicité inter-état ACTIVE/CLAIMABLE, coût wrapper négligeable. Garder.
- **`combat-report.service.ts` `canAccessReport` / `getReportRole` / `hiddenPatchForRole` / `readPatchForRole`** : déjà DRY, pas de duplication à réduire.

## Choix du thème

RS1+RS2+RS3 = même pattern chirurgical dans 2 fichiers (`combat-report.service.ts`, `reinforcement-report.service.ts`), gain de duplication net (-100L brut, +50L helpers), aligné sur `caravan-report.service.ts` qui a déjà `findCaravanEntry`. Vérifiable via smokes existants (`combat-reports-inbox`, `scouting`, `reinforcements`, `caravan`). Aucune migration, aucune balance.

## Changements

- `combat-report.service.ts:37-84` — `getReport` / `deleteReport` / `markReportAsRead`: chaque méthode délègue à un nouveau helper `findAuthorizedCombatReport(userId, reportId, worldId, action)` (assertion membership → findFirst → NotFound/BadRequest guard). Le paramètre `action: 'view' | 'modify' | 'delete'` préserve les 3 phrasings d'erreur d'origine — les opérateurs peuvent toujours grepper les incidents par verbe.
- `combat-report.service.ts:99-135` — `getScoutReport` / `markScoutReportAsRead` / `deleteScoutReport`: idem avec `findAuthorizedScoutReport` (membership → findFirst → NotFound + BadRequest si `scoutUserId !== userId || hidden`).
- `reinforcement-report.service.ts:78-104` — nouveau helper privé `findReinforcementEntry(userId, reportId, worldId, { excludeHidden })` qui miroite `CaravanReportService.findCaravanEntry`. Le paramètre `excludeHidden: true` gate le GET (ne surface pas un soft-deleted) sans casser le comportement de mark/delete qui restent idempotents sur `hidden`. Type retour `ReinforcementEntryWithReport` (NonNullable) évite le `!` fantôme aux callsites.
