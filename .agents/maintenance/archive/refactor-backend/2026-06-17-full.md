### Looks bad but fine

- **combat.worker.ts taille (1772 lignes)** : cohabite ATTACK/REINFORCE/SCOUT/CARAVAN dans un seul `@Injectable`. Découper coûterait 3-4 nouveaux modules pour un gain de lisibilité modeste — la majorité du fichier est en méthodes privées déjà extraites. Non bloquant.
- **conquest.service.ts (644 lignes)** : compact pour ce qu'il fait (open/finalize/interrupt + worldconqueredEvent + capture report). Pas de god-service.
- **village-strategy.service.ts:184–313** : atomic guards bien commentés (configUpdate + crown debit + stock debit avec `updateMany` `gte:`). Pattern de pointe — ne pas toucher.
- **resources.service.ts:48–51** : la récursion après `updateProduction` est volontaire (force read-after-write) ; pourrait être inline mais clair tel quel.
- **applyDefenderLosses** dans combat.worker (113 lignes) : 5 cas distincts (defender vs reinforcement origin × {pending capture noble killed, escort destroyed}) — la complexité est inhérente au domaine, pas extractible.
- **power.service.ts** : pas un god-service, leaderboard / kingdom power / village power bien découpés.
- **rankings.service.ts** : 264 lignes mais flat, helpers `creditGlory` + 3 leaderboards. Pas de duplication réelle.
- **retention.service.ts** : déjà fragmenté en `retention.utils.ts` par PR #109. Bonne ligne de partage.
- **onboarding.service.ts** : déjà fragmenté en `onboarding.utils.ts` par PR #126.

### Thème choisi

**OB1 + OB2 — Extract notification planner from EventOutboxService**.

Pourquoi :
- High severity / Med effort : `event-outbox.service.ts` (853 lignes) est la 3e plus grosse classe, et 78% (≈660 lignes) du contenu sont des `notifyXxx` quasi-identiques.
- Toute addition d'event aujourd'hui = 3 endroits à toucher : schema (`shared/events`), case du switch dispatcher, méthode privée. Avec la table → 1 endroit (le planner).
- Verifiable via `yarn static-check` + suite jest existante (prefetch + payload codec) + 1 nouveau spec unitaire ciblé.
- Pas de mutation DB, pas de behavior change, pas de balance.

## Refactor (Phase 2)

### Plan

1. Créer `event-outbox-notification-planner.ts` (pure) :
   - Types `NotificationRecipient`, `NotificationPlan`, `NotificationPlannerDeps`.
   - Table `PLANNERS: Record<EventKind, Planner>` avec helpers `userByVillage(field)`, `usersByVillages(fields)`, `world(field)`, `directUser(field)`.
   - Cas spéciaux explicites : `village.attacked` (defender sans observerUserId + observer avec), `village.conquered` (newOwner + previousOwner), `village.capture-window-opened/interrupted` (fallback PendingConquest), `village.capture-window-completed` (newOwnerUserId direct), `noble.killed` (attackerUserId direct).
   - `planNotifications(kind, payload, deps): Promise<NotificationPlan[]>`.
2. `event-outbox-notification-planner.spec.ts` :
   - Coverage par "shape" (userByVillage, usersByVillages dedup, world, directUser, village.attacked observer split, village.conquered dual).
3. `EventOutboxService.dispatchEvent` :
   - Remplacer le switch + 25 méthodes par `await planNotifications(kind, payload, deps)` + boucle `notifyUser / notifyWorld`.
   - Garder `recordOutboxEvent` (retention + onboarding) avant.
   - Garder `prefetchVillageUserIds`, `getUserIdByVillage`, `getAttackerUserIdByConquest`, `getUniqueUserIdsByVillages` (utilisés par les deps du planner ; le 4e est devenu inutile et supprimé).
4. Verification : `yarn static-check`, `yarn test:backend`, smokes si touched files réintroduisent un risque (`combat`, `caravan`, `onboarding`).
