# Refactor backend — 2026-06-22 (run 2)

## Thème retenu

**W5 + W6 — `construction.worker.ts` post-tx correctness**

Le callback `$transaction` retourne en early-return dans 3 cas (`building` introuvable, cancellation pré-firing avec `endTime=null`, building déjà au niveau cible) mais les 3 side-effects post-tx (`updateStorageLimit`, `outbox.resourcesChanged`, `crownsService.recalculateOnBuildingChange`) s'exécutaient quand même. Le cas le plus visible : annuler un upgrade WAREHOUSE laissait `resourceStock.maxPerType` être bumpé vers le cap du niveau jamais atteint quand pg-boss firait le singleton.

Inclus collatéral : les 3 try/catch « Don't throw » swallow silencieux (W5) — restent silencieux par design (le build est durable, on ne veut pas retry tout le job) mais loggent désormais un payload structuré (`op`, `buildingId`, `villageId`, `buildingType`, `targetLevel`, `error.stack`) au lieu du seul message libre.

### Looks bad but fine

- Le pattern « catch + log + don't throw » sur les side-effects post-tx reste correct : la mutation `building.level` est déjà committée par le `$transaction`, retry-er tout le job réémettrait `building.completed`. Ne pas pousser ces side-effects DANS la tx (`updateStorageLimit` ouvre sa propre tx, `recalculateOnBuildingChange` lock SELECT FOR UPDATE → dead-lock risk si imbriqué).
- L'utilisation de `BUILDING_TYPES.WAREHOUSE` côté worker au lieu de la string magique `'WAREHOUSE'` aligne sur le reste du worker (qui utilise déjà `BUILDING_TYPES.QUARTER`).
- La pg-boss singleton n'est pas supprimée à `cancel` — choix de design : le worker doit déjà être idempotent (`building.level >= targetLevel`), et delete une singleton mid-flight est plus coûteux côté pg-boss qu'un no-op côté worker.

## Phase 2 — Refactor exécuté

### Worker

`battleforthecrown-backend/src/workers/construction.worker.ts` :

- Extract `applyCompletionInTransaction(data, receivedAt): Promise<boolean>` — encapsule la tx, retourne `true` si la mutation `building.level` est faite, `false` sur early-return (not found / cancelled / already done).
- Extract `runPostCompletionSideEffects(data)` — gated sur le boolean ; n'est appelé que si la tx a vraiment muté.
- Extract `runIsolatedSideEffect(op, data, fn)` — try/catch + log structuré (`op`, `buildingId`, `villageId`, `buildingType`, `targetLevel`, `error.stack`). Centralise la rule « Don't throw » et améliore l'observabilité.
- Constante `PRODUCTION_RECOMPUTE_BUILDINGS: ReadonlySet<BuildingType>` au top du fichier (`WOOD`, `STONE`, `IRON`, `WAREHOUSE`) au lieu de l'array littéral inline.
- Remplacement de la string magique `'WAREHOUSE'` par `BUILDING_TYPES.WAREHOUSE`.

Net : +35L / -27L (185L total), zero changement de comportement happy-path, fix observable du bug W6 dans le cas WAREHOUSE cancelled-mid-flight.
