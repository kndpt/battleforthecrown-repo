# refactor-backend — run 2026-06-15

## Thème : O2 — Extraction des utils onboarding hors du fichier service

### Contexte

`onboarding.service.ts` (408 lignes) suivait le même anti-pattern que `retention.service.ts` avant le run R3 (PR #109) : la classe `@Injectable` était suivie de fonctions exportées et helpers privés au niveau module, mélangeant orchestration NestJS et logique pure (projection d'événements, mapping DTO, ordre des étapes, classification d'erreurs Prisma).

Symétriquement à R3 — qui a sorti `retention.utils.ts` du service pour clarifier les responsabilités — O2 extrait la couche pure d'onboarding dans `onboarding.utils.ts`.

### Changements

| Fichier | Action |
|---|---|
| `src/modules/onboarding/onboarding.utils.ts` | Nouveau — 129 L : `ONBOARDING_INITIAL_REWARD`, `ONBOARDING_STEP_ORDER`, `getOnboardingProjection`, `getNextStep`, `mapOnboardingState`, `isUniqueConstraintError` |
| `src/modules/onboarding/onboarding.service.ts` | 408 → 294 L ; imports nettoyés (drop des payloads d'event, `PrismaOnboardingStep`) ; ré-export `ONBOARDING_INITIAL_REWARD` depuis utils pour compat externe |
| `src/modules/onboarding/onboarding.service.spec.ts` | Import redirigé vers `./onboarding.utils` |

Les helpers DB-bound (`reconcileStateFromFacts`, `isStepSatisfied`, `hasBuildingLevel`, `hasMilitiaTarget`, `hasVictoriousBarbarianAttack`) **restent** au niveau module dans `onboarding.service.ts` : ils prennent une `Tx` Prisma et ne sont pas purs — sortir un `onboarding.reconcile.ts` aurait ajouté de la friction sans bénéfice clair.

### Looks bad but fine

- **combat.worker.ts (1780 L)** : structure orchestration → sub-méthodes (`writeCombatReport`, `creditCombatGlory`, `handleConquestOutcome`, `computeReturn`, `finalizeExpedition`). Pas un god-method à proprement parler. Découper en sous-services nécessiterait un run dédié + suite de tests intégration musclée. Pas dans le scope d'un thème seul.
- **event-outbox.service.ts notify\* duplication** : ~20 méthodes `notify*` suivent toutes le pattern `getUserIdByVillage → if(!userId) return → gateway.notifyUser`. PR #91 modifie exactement ce code (cache batch-scoped). Refactor parallèle conflit → attendre merge PR #91 avant d'extraire un helper `notifyByVillage`.
- **G1** : reconfirmé — séquentiel intentionnel pour tx Serializable, risque deadlock si parallélisé.
- **D4** : reconfirmé — variants `expedition.create` ont assez de champs spécifiques (`reinforcementOriginVillageId`, `reinforcementRecallActorUserId`, `loot`, `defenderKingdomPowerSnapshot`, `attackerKingdomPowerSnapshot`) pour qu'une factory unifiée soit plus dense que le code étalé.
- **U1 (inbox createMany)** : net mais bénéfice perf réel mince (≤2 recipients par expedition). À garder en backlog jusqu'à preuve d'impact.
