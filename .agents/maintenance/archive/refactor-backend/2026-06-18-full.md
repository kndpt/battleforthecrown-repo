- **village-strategy.service.ts:184–313** : atomic guards bien commentés (configUpdate + crown debit + stock debit avec `updateMany gte:`). Ne pas toucher.
- **resources.service.ts:48–51** : la récursion après `updateProduction` est volontaire (force read-after-write).
- **power/rankings/retention** : tailles raisonnables, helpers déjà extraits (PR #109 pour retention, PR #126 pour onboarding).
- **event-outbox.service.ts (210 L)** : déjà allégé via PR #134 (planner).

### Thème choisi

**D4 — Centralize `buildings.find((b) => b.type === X)` into shared helpers**.

Pourquoi :
- 11 sites identiques (6 fichiers) ; le pattern existait déjà comme méthode privée dans `village-strategy.service.ts` mais n'était pas réutilisé ailleurs.
- Faible risque : pure logic, pas de migration, pas de behavior change.
- Verifiable via `yarn static-check` + Jest backend + smokes existantes sur tous les modules touchés (combat, resources, gameplay).
- Petit, bien borné, contribue à la lisibilité sans changer la sémantique.
