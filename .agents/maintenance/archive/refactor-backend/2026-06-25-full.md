## Refactor — run 2026-06-25

**Thème** : consolider l'insertion du row `Expedition` entre `initiateAttack` /
`initiateScout` / `initiateReinforce` dans `combat.service.ts`.

**Changement** : nouveau helper privé `createOutboundExpedition(tx, params)` qui
calcule la timing (`calculateExpeditionTiming`) puis insère le row Expedition
avec les champs de base (`worldId`, `attackerVillageId`, `kind`, `targetKind`,
`targetRefId`, `targetX/Y`, `units` encodés, `status: 'EN_ROUTE'`, `departAt`,
`arrivalAt`, `outboundTravelMs`). Les variant-only fields (attack: kingdom
power snapshots; reinforce: `reinforcementOriginVillageId`) passent par
`extraData?: Partial<Omit<ExpeditionUncheckedCreateInput, /* base */>>` — le
type Omit empêche l'override silencieux d'un champ de base.

**Conservé délibérément hors helper** : `createOutboxEvent`, side-effects PvP
(shield break), `scheduleResolution`. Préserve l'ordre exact tx-local: create →
outbox → side-effect → schedule. Aucun risque nouveau de job pg-boss orphelin.

**Diff** : 1 fichier, +112 / -87 (net +25L). La verbosité du helper (TSDoc +
generics Omit) coûte ~75L mais centralise une vérité unique : ajouter un 4ème
type d'expédition outbound = 1 endroit à modifier, pas 3. Tous les champs
métier alignés (le typo entre `now` non-utilisé dans scout/reinforce vs. attack
disparaît implicitement).

**Verification** : `yarn static-check` ✅. `yarn test:backend` 508/508 ✅. Smokes
ciblés ✅ : `combat-attack`, `scouting`, `reinforcements`, `pvp-newbie-shield`,
`pvp-power-third-guard`, `recall-en-route`, `caravan` — 25 tests, 0 régression.

touchées. Migration N/A.
