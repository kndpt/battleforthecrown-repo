### Looks bad but fine

- **`RetentionService.getSummary` triple `Promise.all` sur `dailyCard.findMany/findFirst` + `getActiveOyez`** — même client Prisma, 3 requêtes independantes, non foldable en une jointure.
- **`map-marker.service.ts:76` advisory lock via `pg_advisory_xact_lock`** — race count-vs-cap, préserve l'invariant CAP. Volontaire.
- **`friendship.service.ts:234-240` `SELECT ... FOR UPDATE` raw** — verrouillage bilatéral pour double-cap, seule alternative Prisma.
- **`rankings.service.ts:73-90` 24h diminishing returns** — l'aggregate est nécessaire pour bornes précises ; pas foldable dans le `create`.
- **`world.service.ts:222-238` touchUserMembership 2 round-trips (avant refactor)** — la validation du statut passe AVANT l'update ; on ne peut pas skip le premier read.

## Choix du thème

`WorldService` — pattern d'agrégation `count/groupBy` séparé, 4 méthodes concernées, `_count` filtrable Prisma 6 résout tout, aligné avec `getPublicWorlds` déjà en place. Vérifiable via smokes existants (`worlds public smoke`, `world membership smoke`, `world-archive lifecycle smoke`). Aucune migration, aucune balance.

## Changements

- `world.service.ts:32-42` — `getWorlds`: 2 requêtes (`findMany worlds` + `groupBy memberships`) → 1 requête `findMany` avec `include: { _count: { select: { memberships: true } } }`. Destructuration `{ _count, ...world }` dans le map.
- `world.service.ts:44-68` — `getWorldDetails`: 2 requêtes (`findUnique world` + `count worldMembership`) → 1 requête `findUnique` avec `_count.memberships` dans le `select`.
- `world.service.ts:161-206` — `getUserMemberships`: 2 requêtes (`findMany memberships` + `groupBy villages`) → 1 requête `findMany` avec `_count` filtré `{ villages: { where: { userId } } }` sur la relation `world`. Suppression `villageCounts`/`countByWorld` Map intermédiaire.
- `world.service.ts:208-249` — `touchUserMembership`: 3 requêtes (`findUnique` + `update` + `count villages`) → 2 requêtes (`findUnique` validation status, puis `update` avec `_count.villages` filtré `{ where: { userId } }` sur `world`). L'initial `findUnique` reste pour valider le statut AVANT le side-effect (`lastLoginAt`).
